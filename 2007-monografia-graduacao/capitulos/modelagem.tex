    \UTFPRsection[sec:metodologia]{METODOLOGIA}

    Neste sistema, o processo de desenvolvimento adotado é baseado no
    \UTFPRsigla{PDSU}{Processo de Desenvolvimento de \emph{Softwares}
    Unificado}, denotado nesse trabalho como
    \UTFPRsigla{UP}{\emph{Unified Process}}. Este é usado em
    conjunto com a \UTFPRsigla{UML}{\emph{Unified Modeling Language}}, e
    o paradigma de orientação a objetos.

    A Figura \ref{fig:work} representa graficamente as fases (concepção,
    elaboração, construção e transição), as tarefas do processo
    (requisitos, análise, projeto, implantação e testes), e a quantidade
    de trabalho estimado para tarefas específicas em cada fase.

    \UTFPRfigura[\cite{arlow:2003}]{width=15.5cm,height=9cm}{metodologia/fases_workflows.jpg}{Quantidade
    de trabalho aplicada em cada workflow nas fases do UP}{fig:work}

    O UP é iterativo e incremental, ou seja, para cada uma de suas
    fases: concepção, elaboração, construção e transição; existe uma
    quantidade de trabalho necessária para cada fluxo: requisitos,
    análise, projeto, implantação e teste. De acordo com a necessidade
    do projeto pode-se estabelecer um número de iterações para uma fase.
    O UP é, ainda, baseado em casos de uso \cite{livro:up}.

    A concepção estabelece a viabilidade e escopo do sistema, define e
    estrutura uma arquitetura, identifica os riscos, esclarece questões
    de custo-benefício. A elaboração deve coletar os requisitos, visando
    recolher o máximo de informação útil e solidificar a arquitetura.
    Essa fase visa o desenvolvimento da maioria dos casos de uso e a
    especificação das metas de qualidade. A construção desenvolve uma
    versão operacional do sistema que possa ser entregue à usuários
    beta. Nessa etapa constrói-se a maior parte dos modelos de projeto,
    análise, instalação, implantação e teste. E, por fim a transição, a
    fase final do projeto, quando é realizada a entrega do sistema aos
    usuários beta com todas as funcionalidades finalizadas (e
    futuramente a todos os usuários). A divulgação do \emph{software}, e
    preparação dos usuários também fazem parte dessa fase
    \cite{livro:up}.

    Cada uma dessas fases é dividida em sub-processos, chamados fluxos de
    trabalho ou \emph{workflows}. Cada fluxo de trabalho tem um tema
    técnico específico, enquanto as fases constituem divisões
    gerenciais, caracterizadas por atingirem metas bem definidas
    \cite{livro:engenharia}. Pode acontecer uma ou mais iterações por
    fase do processo, cada iteração gerará uma gama de artefatos
    específicos dos fluxo de trabalho.

    Neste trabalho, com o uso desta metodologia e baseado nos \emph{workflows}
    citados, foram gerados os seguintes artefatos para fins de
    documentação:

        \begin{itemize}

        \item requisitos: fluxo que visa obter um conjunto de requisitos de um produto, aprovado
        pelo cliente e desenvolvedor \cite{livro:engenharia}.

        \begin{itemize}
            \item levantamento dos requisitos funcionais e
            não-funcionais;

            \item levantamento dos atores e casos de uso;

            \item levantamento dos riscos;

            \item especificação dos casos de uso;

            \item diagrama de caso de uso.
        \end{itemize}

        \end{itemize}

        \begin{itemize}
        \item análise: fluxo cujo objetivo é detalhar, estruturar, e validar os requisitos, de forma
        que esses possam ser usados como base para o planejamento detalhado \cite{livro:engenharia}.

        \begin{itemize}
            \item refinamento dos casos de uso;

            \item especificação de casos de uso;

            \item diagrama de seqüência.

        \end{itemize}

        \item projeto: fluxo cujo objetivo é formular um modelo estrutural do produto que sirva de
        base para a implementação \cite{livro:engenharia}.

        \begin{itemize}
            \item diagrama de classe de projeto;

            \item \UTFPRsigla{MR}{Modelo Relacional};

            \item diagrama de pacotes;

            \item perguntas freqüentes (Apêndice \ref{ape:faq}).

        \end{itemize}

        \item implementação: fluxo cujo objetivo é realizar o projeto em termos de componentes de
        código \cite{livro:engenharia}.

        \begin{itemize}
            \item diagrama de implantação;

            \item diagrama de componentes.
        \end{itemize}

        \item teste: fluxo cujo objetivo é verificar os resultados da implementação
        \cite{livro:engenharia}.

        \begin{itemize}
            \item gerar casos de teste das principais funcionalidades do
            sistema.
        \end{itemize}

        \end{itemize}

    \UTFPRsection[sec:uml]{UML}

    A UML é uma linguagem gráfica para visualização, especificação,
    construção e documentação de artefatos de sistemas complexos de
    \emph{software}. A UML proporciona uma forma padrão para a preparação de
    planos de arquitetura de projetos de sistemas, incluindo aspectos
    conceituais, tais como: processos de negócios e funções do
    sistema. Além de ítens concretos como as classes escritas em determinada
    linguagem de programação, esquemas de bancos de dados e componentes
    de \emph{software} reutilizáveis \cite{livro:uml}.

    Neste trabalho, a linguagem UML serviu para a documentação dos
    artefatos descritos na Seção \ref{sec:artefatos}.

        \UTFPRsubsection{Diagrama de implantação}

        O diagrama de implantação determina as necessidades de hardware do
        sistema, as características físicas como servidores, estações,
        topologias e protocolos de comunicação, ou seja, todo o aparato
        físico sobre o qual o sistema deverá ser executado
        \cite{td:shido}.

        \UTFPRfigura{width=13cm}{diagramas/DiagramaDeImplantacao.jpg}{Diagrama de implantação}{fig:diagramadeimplantacao}

        Na Figura \ref{fig:diagramadeimplantacao}, pode-se observar a
        interação da máquina cliente com o servidor \emph{web} através do
        protocolo HTTP pela internet. O servidor \emph{web} abriga os dois
        sistemas e acessa eventualmente dois bancos de dados, um deles
        necessários para computar os registros do sistema OriAnt e outro
        para armazenar os dados do \emph{website} hospedeiro.

        Para a implantação do sistema é necessário que o administrador do
        \emph{website} hospedeiro copie todo o conteúdo de seu site para
        dentro da pasta PHP, localizada dentro da pasta "oriant"; crie um
        novo banco de dados PostgreSQL, e carregue as tabelas do arquivo
        "oriant/bd/sql/sql.sql"{}. Ainda é necessário que se edite o arquivo
        de configuração "oriant/conf/config.php"{}, alterando as informações
        para conexão no banco de dados (nome, servidor, usuário e senha).
        Feito isso, já é possível utilizar o sistema OriAnt.

        \UTFPRsubsection{Diagrama de componentes}

        O diagrama de componentes está amplamente associado à linguagem de
        programação que será utilizada para desenvolver o sistema modelado.
        Esse diagrama representa os componentes do sistema quando este for
        implementado em termos de módulos de código-fonte, bibliotecas,
        formulários, arquivos de ajuda, módulos executáveis etc, e determina
        como esses componentes estarão estruturados e interagirão para que o
        sistema funcione de maneira adequada \cite{td:shido}.

        \UTFPRfigura{width=17cm}{diagramas/DiagramaDeComponentes.jpg}{Diagrama de componentes}{fig:diagramadecomponentes}

        Na Figura \ref{fig:diagramadecomponentes}, os pacotes:
        "\emph{SpaceBrain}", "\emph{Frameworks}"{} e "Classes"{} representam
        as classes mostradas na Seção \ref{subsec:diagramadeclasses}. Os
        componentes do pacote "javascript"{} são utilizados pela página
        principal localizada no pacote "oriant"{} e estão descritos na Seção
        \ref{subsec:javascript}. O pacote "cont"{} possui os arquivos para:
        mapeamento das tabelas, tratamento de erros e configuração do
        sistema, descritos respectivamente nas Seções \ref{sec:mapeamento},
        \ref{sec:erros} e \ref{sec:configuracao}. O pacote "HTML"{} e
        "CSS"{} são unidos e são formados pelos arquivos para elaboração dos
        \emph{templates} de interface do sistema. O pacote "lang"{} é
        composto pelos arquivos de tradução do sistema. O pacote "ajax"{}
        abriga os arquivos para implementação das funcionalidades em AJAX. O
        pacote gerenciar é formado por um arquivo genérico para
        gerenciamento dos registros das tabelas do banco de dados, já
        detalhado na Seção \ref{subsubsec:gerenciamento}. O pacote "lib"{}
        contém um arquivo com várias pequenas funcionalidades utilizadas
        pelo sistema, como geração de data, tratamento de strings, entre
        outros, e o pacote "php"{} abriga todo o \emph{website} hospedeiro.
